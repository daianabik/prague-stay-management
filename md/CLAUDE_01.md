## Key findings from the EDA

**listings** — 10 834 rows × 79 columns:
- `price` is stored as a string (`$2,272.00`). The `$` symbol is an Inside Airbnb export artefact — values are already in CZK, confirmed by the median (~2 152 CZK/night).
- ~12 % of price values are missing.
- Three columns are 100 % NaN: `license`, `neighbourhood_group_cleansed`, `calendar_updated`.
- `price_czk` weakly correlates with `accommodates` (r = 0.09), meaning larger apartments are not necessarily more expensive — relevant for KPI #3.

**calendar** — 3 954 411 rows × 7 columns:
- `price` and `adjusted_price` are both 100 % NaN. The calendar is only useful for availability (`available`: `t`/`f`).
- Some listings show blocks of `available = f` up to 366 consecutive days — likely long-term rentals or owner blocks, not real bookings.

**metro_stations** — 61 rows × 5 columns. Clean, no issues.

## Conventions used in this project

- `RAW = '../data/raw/'` — all raw file paths use this constant (paths are relative to `notebooks/`)
- Plot theme: `sns.set_theme(style='whitegrid', palette='muted')`, default figsize `(12, 5)`
- Warnings suppressed via `warnings.filterwarnings('ignore')`
- Use `listings['price_czk']` (float, already parsed) — not `listings['price']` (raw string with `$`)
- Use `calendar['occupied']` (0/1, derived from `available == 'f'`) for any occupancy calculation

## What to work on in notebook 02

The summary section (Section 6) lists seven concrete issues to resolve. The short version:

1. Parse `listings.price` — strip `$`/`,`, convert to float
2. Impute missing prices by median grouped by neighbourhood + room type
3. Drop columns with more than 50 % missing values
4. Use `calendar.available` only — ignore `calendar.price`
5. Keep price outliers for now; trim at the 99th percentile in analysis
6. Generate `bookings.csv` and `expenses.csv` as synthetic datasets
7. Calculate Haversine distance from each listing to the nearest metro station

All of this is implemented in `02_data_preparation.ipynb`.
